# Heltec V4.3 Notes

## Problem observed

- A real Heltec v4.3 board running the MQTT firmware reported itself as `Heltec V4`, not `Heltec V4.3`.
- Transmit errors were observed and suspected to come from the wrong FEM path being selected at boot.

## Root cause

Two things were wrong in the MQTT firmware fork:

1. **Missing runtime FEM detection** — the upstream autodetect code in `LoRaFEMControl::init()`
   was absent, so the firmware always selected the GC1109 (v4) path.

2. **DIO2 RF switch misconfiguration** — the base `Heltec_lora32_v4` env sets
   `SX126X_DIO2_AS_RF_SWITCH=true` at compile time. `CustomSX1262::std_init()` applies this
   unconditionally. For KCT8103L (v4.3), DIO2 must NOT be the RF switch — the MCU drives CTX
   on GPIO5 directly. Even with runtime FEM detection in place, this compile-time flag caused
   the radio to be misconfigured on v4.3 hardware.

## Fix (two parts)

### 1. Runtime FEM autodetect in `LoRaFEMControl::init()`

Added the upstream autodetect logic (matching stock Heltec firmware). At cold boot:

1. Releases the RTC hold on `GPIO2` (in case of deep-sleep wake).
2. Configures `GPIO2` as `INPUT` and waits 1 ms for the line to settle.
3. Reads the pin:
   - `HIGH` → `KCT8103L` path (v4.3)
   - `LOW` → `GC1109` path (v4)

Compile-time overrides are also available for boards where autodetect is unreliable:
- `HELTEC_V4_FORCE_KCT8103L_PA` — always use KCT8103L / v4.3 path
- `HELTEC_V4_FORCE_GC1109_PA` — always use GC1109 / v4 path

### 2. Runtime DIO2 override in `target.cpp`

After `radio.std_init()`, DIO2 is re-configured based on the FEM detected at runtime:

```cpp
radio.setDio2AsRfSwitch(board.loRaFEMControl.getFEMType() != KCT8103L_PA);
```

This overrides whatever the compile-time `SX126X_DIO2_AS_RF_SWITCH` define set. No changes
to mainline `src/` files — the fix is confined to `variants/heltec_v4/target.cpp`.

## Dedicated v4.3 targets

`heltec_v4_3_repeater` and `heltec_v4_3_tft_repeater` still exist as an explicit fallback for
boards where GPIO2 doesn't autodetect reliably. They set `HELTEC_V4_FORCE_KCT8103L_PA=1` and
`SX126X_DIO2_AS_RF_SWITCH=false` at compile time — the runtime override in `target.cpp` is
consistent with this and sets DIO2 to `false` as well.

## MQTT dynamic builds

The dynamic MQTT build (`bin/build-meshcore-mqtt-observer.sh`) supports both variants:
- `MESHCORE_MQTT_BASE_ENV=heltec_v4_repeater` — autodetect at boot (works on both v4 and v4.3)
- `MESHCORE_MQTT_BASE_ENV=heltec_v4_3_repeater` — force KCT8103L (if autodetect is unreliable)

## CLI behavior to expect

- `heltec_v4_repeater` MQTT firmware autodetects at boot. A v4.3 board should report:
  - `Heltec V4.3 OLED`
  - or `Heltec V4.3 TFT`
- If autodetect fails, use `heltec_v4_3_repeater` as the base env.
